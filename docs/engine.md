## 1. 「koe」の再定義：連結合成エンジン

`koe` は、物理モデリング生成エンジンではなく、「UTAU方式の連結合成（Concatenative Synthesis）をWASM上で最適化した高速再生エンジン」と定義します。

### 技術的な課題と解決策

* **課題:** モバイル端末でのリソース制限と、リアルタイム再生時の計算負荷の変動。
* **解決策:**
* **コンバーター:** 全音素を単一フォーマット（48kHz/16bit/Mono）へ正規化し、メモリレイアウトを最適化。
* **ランタイム:** WASM内で「リサンプリング（ピッチ変更）」と「クロスフェード（連結）」のみを定常的に処理し、計算負荷を一定にする。



---

## 2. 処理系（Runtime Engine）の設計

WASM側で実行するメインの処理ループを以下のように設計します。

### A. 音素管理（Memory Layout）

`voice.bin` から読み込まれたRaw PCMデータは、WASMの線形メモリ上で「音素配列（Phoneme Table）」として保持します。

```text
[WASM Linear Memory]
+-------------------+-------------------+-------------------+
|  Phoneme A (Data) |  Phoneme Ka (Data)|  Phoneme Su (Data)| ...
+-------------------+-------------------+-------------------+
(Metadata: Pointer, Length, Offset)

```

* **計算の均一化:** 再生時の計算は、ポインタを指定して `Raw PCM` を読み取り、目的のピッチに合わせてインデックスを歩進（リサンプリング）するのみ。これにより、どんな音素を再生しても計算コストは一定です。

### B. オーディオ処理（AudioWorklet）

`process()` 内で毎フレーム以下の処理を完結させます。

1. **Read:** `manifest` から現在の音素の `start` / `overlap` 情報を取得。
2. **Resample:** 指定されたピッチに応じたレートで波形を補間（Linear interpolation等）。
3. **Overlap/Crossfade:** 前後の音素の接合部分に対し、定義された `overlap` 期間でエンベロープ（音量カーブ）を掛け合わせる。
4. **Write:** `outputs[0]` に書き出し。

## 3. コンバーターの役割（データ最適化）

ランタイムの計算負荷をゼロにするために、コンバーター側で前処理を徹底します。

* **oto.ini の解釈:** UTAUの `oto.ini` は記述がバラバラですが、これを全て「絶対時間（ms）」から「サンプル数」に変換した `manifest.json` を生成します。
* **正規化:**
* すべてのWAVを 48kHz/16bit/Mono に統一。
* これにより、エンジン側での複雑なサンプリングレート変換処理を回避。


* **順序の並び替え:** 頻出する音素を `voice.bin` の前方に配置する等の最適化も可能です（ロードの優先度制御）。


* **文字コードの正規化:** Node.jsの `iconv-lite` を使い、`oto.ini` をShift-JISからUTF-8に強制変換してパースします。
* **パスの抽象化:** ユーザー環境依存の絶対パスや、OSごとの区切り文字（`\` / `/`）を、リポジトリ内相対パスとして解決します。
* **オーディオのリサンプリング:** UTAU音源は「22kHz/8bit」から「96kHz/24bit」まで混在しています。これらをすべて「48kHz/16bit/Mono（PCM）」に強制リサンプルします。これにより、WASMエンジン側ではリサンプラーを実装する必要がなくなり、再生ロジックが極限まで軽量化されます。
* **妥当性チェック:** 参照しているWAVファイルが存在しないエントリ、あるいは `oto.ini` の記述が矛盾しているエントリは、ログを出して除外するか、安全な値で埋めるルールを適用します。

## 4. この設計におけるモバイルファーストの利点

* **計算コストの固定:** 物理モデル合成のように複雑なフィルタ計算をリアルタイムで行わないため、CPU負荷がスパイクせず、バッテリー消費が安定します。
* **ゼロコピー:** `voice.bin` は `SharedArrayBuffer` や `Transferable` で渡せば、メインスレッドとエンジン間でコピーが発生せず、低メモリで動作します。

#### モバイルファーストの要諦

モバイルブラウザ（特にiOS）では、以下の制約が厳しいです。

1. **メモリ制限:** 全音源をメモリに乗せると即クラッシュします。
* **対策:** `voice.bin` をストリーミング（Range Request）で取得し、必要な部分だけを `AudioBuffer` にデコードします。


2. **デコード遅延:** 毎回WAVをデコードするのは重いです。
* **対策:** `voice.bin` は可能な限りRaw PCM（未圧縮）で出力します。ブラウザ側でのデコード負荷をゼロにします。


3. **バッテリー消費:**
* **対策:** WASMエンジンによる処理を `AudioWorklet` 内に閉じ込め、メインスレッド（UI描画）への影響を皆無にします。
